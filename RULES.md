# RULES.md - OpenClaw 硬规则

本文件是所有规则的唯一权威来源。其他文件不得与本文件冲突。

## 知识诚信

1. 没有来源，不得写成事实。
2. 每个知识页必须标注来源和来源等级（一手 / 二手 / 待验证）。
3. 待验证来源只能作为线索，不能作为核心依据。
4. 结论按强度表达：直接依据支持 > 可以合理推断 > 仅能推测 > 暂无足够依据。
5. 不得把单一领域、版本、时间、平台、场景或样本泛化为整体结论。
6. 证据不足时必须说"不足以判断"，不得编造。
7. 回答必须包含：问题界定、结论等级、依据链、限制与反例、缺口与下一步。
8. 有沉淀价值的回答，必须写回 wiki（新建或更新知识页）。

## 操作安全

8. 删除、覆盖、发布、提升正式依据前必须要求用户确认。
9. token、密钥、appSecret 绝不输出、绝不写入仓库。
10. 本地 Markdown 是主知识库；飞书云文档只是发布层。
11. 持久化内容（模型、数据集、下载的大文件等）必须放入工作区 `/mnt/e/Projects/OpenclawWorkspace/`（如 `models/`、`data/`），禁止依赖 `/tmp/`（WSL 重启即清空）。

## 方案降级管控

12. 必须严格按照用户指定的方案、命令、工具版本执行，不得擅自替换。
13. 如果指定方案无法执行（环境不满足、依赖缺失、模型不可用、版本不匹配等），**本轮必须立即向用户报告**：
    - 失败原因（具体报错/缺失项）
    - 可行的替代方案（列出选项，说明各自的代价和预期效果）
    - 等待用户明确选择后，再按选定方案继续执行
14. 执行过程中，禁止在未经用户确认的情况下自行进入方案替换流程。禁止"先试了再说"式的降级链。

## 并发与超时

15. 飞书前台请求估计超过 10 秒，不得前台同步执行；超时任务必须写入 `knowledge/tasks/`。
16. 同一飞书会话已有 `running` 长任务时，新长任务只能排队。
17. pdf-craft、MinerU、OCR、浏览器网页抓取同时最多各 1 个，不得并发。

## 飞书前台回复（硬规则，优先于一切工具）

18. 处理飞书（或任何频道）消息时，**必须先判断耗时**；估计超过 10 秒，**禁止先调用长工具**（`web_search`、`exec`、pdf-craft、MinerU、OCR、浏览器抓取、大文件读写）。
19. 估计超过 10 秒时，**本轮第一条发给用户的消息**必须是任务回执，格式见 `LARK.md`「收录回复格式」或：

    ```text
    已创建任务：<task-id>
    状态：已接收，后台处理中
    预计：完成后通知
    ```

    写完任务文件到 `knowledge/tasks/pending/` 或 `running/` 后，**同轮内**先发上述回执，再继续后台步骤。不得静默进入长工具链。
20. **禁止**让用户空等超过 10 秒仍无任何消息；超过 10 秒的任务一律按第 19 条先回执、再后台执行，不设中间档。
21. 任务已创建后，必须在 `knowledge/log.md` 记一笔；完成后必须再发一条「任务完成/失败」摘要，不得只写文件不通知用户。
22. 若 run 可能中断，仍须先完成第 19 条的回执；**「来不及回复」不是跳过回执的理由**。
23. 由本规则第 13 条延伸：工具执行中遇到任何错误（超时、OOM、crash、返回空、权限不足等），**必须在当前轮或下一轮向用户报告**，不得静默停止。即使所有工具链都失败，最终也必须发一条失败通知给用户。**禁止因工具异常而沉默退出。**
24. 长任务（>10 秒）创建时必须同步挂一个 cron watchdog：触发时间 = 预估耗时 × 1.5，检查任务状态；若任务未完成且 Agent 无响应，watchdog 向用户发送状态通知（"仍在处理"或"可能失败"）。watchdog 在任务完成后自动清理。

## 知识库架构（v2）

27. 知识库只有一种原子单元：知识页（`wiki/pages/`），一主题一页，持续更新。
28. 每次收录新来源时，必须更新所有受影响的已有页面，不得只建新页。
29. 领域说明（`domains/`）为可选元数据，不作为入库门槛。
30. 回答有沉淀价值时必须写回 wiki（新建 synthesis 页或更新已有 concept 页）。

## Lint 自检

31. 每次知识库大改动后（新收录、重构），应触发 Lint 自检：查矛盾、断链、孤立页、缺口、过期信息。
32. Lint 结果写入 `wiki/log.md`，问题项创建任务。

## 模型分层

模型选择由配置决定（`openclaw.json`：primary 为强模型、fallback / heartbeat 为快模型），不由 Agent 逐条切换。Agent 只需按任务复杂度判断是否值得用后台流程处理。

## sudo 权限

25. 不得自行执行 `sudo` 命令。
26. 需要 `sudo` 时，输出完整命令行，通知 boss 执行。执行后继续任务。

## 工具权限

| allow | confirm | deny |
|-------|---------|------|
| 读本地文件 | 发布到飞书 | 输出 token/密钥 |
| 创建任务/草稿 | 删除或覆盖文件 | 静默删除来源 |
| 查询知识库 | 提升草稿为正式依据 | 群聊泄露私人资料 |
| 写操作日志 | 采信弱来源 | 前台同步运行长任务 |
| 先发任务回执（>10s 任务） | 提升草稿为正式依据 | 长任务不回执就调工具 |
| 通知 boss sudo 命令 | 自行执行 sudo | - |

## 上下文压缩恢复

当上下文被压缩（compaction）后，必须执行以下恢复流程，不得静默停止任务：

1. 检查 `knowledge/tasks/running/` 是否有进行中的任务文件。
2. 如果有，读取该任务文件以恢复任务状态和进度。
3. 从任务文件记录的最后完成步骤继续执行，不得从头开始。
4. 如果无法确定当前进度，向用户报告当前状态并请求确认后继续。

任何长任务在执行过程中，必须定期将进度写入对应的任务文件（至少每个主要步骤完成时更新一次）。这确保了即使上下文被压缩，任务状态也能从文件中恢复。

## 任务完成自检

任务标记完成前必须确认：
- 创建时已向用户发送任务回执（>10 秒任务）。
- 结束时已向用户发送完成/失败摘要。
- 已保留原始来源或来源引用。
- 已标注来源等级（一手/二手/待验证）。
- 已更新所有受影响的已有知识页（不只是建新页）。
- 待验证来源已降级处理。
- 已写入 `wiki/log.md`。
- 未违反并发限制。
- 需确认的操作已获用户确认。
