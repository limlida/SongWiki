# KNOWLEDGE.md - 知识库工作流

> 核心理念来自 Karpathy LLM Wiki：知识在录入时编译、页面持续更新、LLM 做全部簿记。这是一个活的知识有机体，不是档案馆。

## 三层架构

```
raw/          原始来源，不可变，LLM 只读
    │
wiki/         LLM 全权维护的知识页。每次加源、每次好回答，都更新相关页
    │
Schema        本文件 + RULES.md + SOUL.md。定义规则，与 LLM 共同演进
```

## 目录结构

```text
knowledge/
├─ raw/            # 原始来源（PDF、图片，不可变）
├─ converted/      # 转换成品（<tool>/<task-id>/）
├─ sources/        # 来源摘要（一手/二手/待验证 + 一句话理由）
├─ wiki/           # 知识本体
│  ├─ index.md     #   分类目录（entity/concept/synthesis）
│  ├─ log.md       #   追加式操作日志
│  └─ pages/       #   知识页（一主题一页，平铺，持续更新）
├─ domains/        # 领域元信息（可选）
├─ tasks/          # 任务状态
├─ reports/        # 查询报告
├─ templates/      # 模板
└─ inbox/          # 飞书临时落地
```

## 知识页

知识库只有一种原子单元：**知识页**。一个主题一页，持续更新。用 frontmatter `type` 区分类型，不建子目录。

```yaml
---
id: page-<slug>
title: <标题>
type: entity | concept | synthesis
tags: [tag1, tag2]
sources: [src-xxx, src-yyy]
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: draft | reviewed | stale
---

# <标题>

## 概述
一句话说清这是什么。

## 核心内容
结构化知识。每次加新源在此补充、修正或标注矛盾。

## 来源
- src-xxx（一手）：<具体出处>
- src-yyy（二手）：<具体出处>

## 关联页面
- [[page-xxx]]：<关联说明>

## 限制与争议
- 标注矛盾、版本差异、覆盖盲区
```

| 类型 | 说明 | 例子 |
|------|------|------|
| `entity` | 具体实体 | 汝窑、苏轼、汴京 |
| `concept` | 抽象概念 | 坊市制度、还原焰、理学 |
| `synthesis` | 跨源综合 | 宋代娱乐活动、陶瓷彩绘演进史 |

## 三大操作

### Ingest（收录）

每次加新源：

1. 判断：涉及哪些已有知识页？需要新建哪些页？
2. **更新已有页**：补充、修正、标注矛盾
3. 新建页（仅对尚不存在的主题）
4. 创建/更新来源摘要（一手/二手/待验证 + 一句话理由）
5. 更新 wiki/index.md
6. 追加 wiki/log.md
7. git commit：`ingest: <来源标题> -- <触达页面数> pages`

> 一个源可能触及 5-15 个已有页面。不要只建新页，要刷新旧页。

**降级**：源 < 5000 字且不触发新关联 → 自动处理，只建 sources/ 摘要，不发讨论。

**书**：先喂目录建结构页，再逐章对话式 Ingest。骨架层书必须对话。

**300 本书分层**：

| 层级 | 处理方式 |
|------|---------|
| 骨架层（10-20 本核心书） | 逐章对话式 Ingest，人全程参与 |
| 填充层（相关但不核心） | 喂目录+外部摘要，LLM 更新已有页面 |
| 标注层（边缘相关） | 只记书名+一句话到 index.md |
| 待深入层 | 不处理，等需要时再说 |

### Query（查询）

1. 读 wiki/index.md 找相关页面
2. 直接匹配 → 读页面 → 综合回答
3. 无直接匹配 → **编织模式**：扫描全索引 → 定位碎片 → 逐页阅读 → 合成
4. 每个断言必须标注源页面
5. 按专业回答协议输出：问题界定 → 结论等级 → 核心回答 → 依据链 → 限制与反例 → 缺口与下一步 → 意外发现（如有）

**回答回填**：综合 ≥3 页且形成新洞察 → 建议保存为 synthesis 页。格式：

```yaml
type: synthesis
derived_from: [page-id-1, page-id-2, page-id-3]
```

### Lint（认知审计）

定期触发，检查六维：

- 新鲜度：confidence:high + last_verified > 90 天 → 标记
- 暗概念：wikilinks 引用 ≥3 次但无独立页 → 提议建新页
- 矛盾：冲突声明对 → 列出，不消解
- 断链 / 孤立页
- 覆盖报告：查询频率 vs 源分布，找薄弱领域
- 缺口：源中提及但 Wiki 无覆盖的主题

## 来源评分

| 等级 | 含义 | 例子 |
|------|------|------|
| **一手** | 原始文献、官方文档、考古报告 | 窑址发掘报告、清宫档案 |
| **二手** | 研究专著、整理著作、评测 | 有泉《陶瓷简史》 |
| **待验证** | 网页、聊天、出处不明的说法 | 论坛帖子、口述 |

评分只需等级 + 一句话理由。待验证来源只能作为线索，不能作为核心依据。

## 回答格式

每个正式查询回答必须包含：

```markdown
## 问题界定
## 结论等级        ← 直接依据支持 / 可合理推断 / 仅能推测 / 暂无足够依据
## 核心回答
## 依据链
## 限制与反例
## 缺口与下一步
## 意外发现（如有）
```

证据不足时必须说「不足以判断」，不得编造。

## Schema 演进

AGENTS.md / KNOWLEDGE.md / SOUL.md / RULES.md 不是一次性写好的蓝图，是在对话中长出来的。

```text
LLM 在操作中遇到 Schema 未覆盖的情况
  → LLM 主动报告：
      「我注意到 XX 在现有 Schema 里没有规则。
        建议在 YY.md 中加入：<规则草案>。
        原因：<实际案例>。要不要加？」
  → boss 确认 → LLM 修改对应文件
  → git commit: 「schema: YY.md -- XX 规则 (trigger: <触发原因>)」
```

LLM 不能自行修改 Schema，必须经过提议 → 确认 → 执行。

## 有机目录演化

wiki/pages/ 初始平铺。某 type 页面 > 10 → LLM 提议建子目录或导航索引页 → boss 决定 → LLM 执行迁移 + git commit。

## 文档转换

| 文档类型 | 工具 |
|----------|------|
| 扫描书 / 古籍影印 | MinerU（vlm-auto-engine 后端） |
| 复杂版面 / 表格密集 | MinerU |
| 带文本层 PDF | 先抽文本，判断是否需 OCR |

转换产物保留在 `converted/<tool>/<task-id>/`。
