# KNOWLEDGE.md - 知识库工作流（v2：活的知识有机体）

> 重构于 2026-06-02。核心理念来自 Karpathy LLM Wiki：知识在录入时编译、页面持续更新、每次加源自动刷新相关页、LLM 做全部簿记。

## 核心理念

| 旧（v1 学术档案馆） | 新（v2 活知识有机体） |
|---------------------|----------------------|
| 四层：依据卡→条目→综合页→领域 | 两层：知识页 + 聚合页 |
| 页面一次性创建，之后冻结 | 每次加新源，更新所有受影响页面 |
| 领域是入库前提 | 自由生长，跨域天然关联 |
| 六维评分 | 三档：一手 / 二手 / 待验证 |
| 无自检 | Lint 定期扫矛盾/断链/缺口 |
| 回答不归档 | 好回答写入 wiki |
| Schema 固定 | Schema 与 LLM 共同演进 |

## 三层架构

```
原始来源（raw/）       — 不可变。LLM 只读不修改。
    │
知识页面（wiki/）      — LLM 全权维护。每次加源、每次好回答，都更新相关页。
    │
Schema（本文件+RULES） — 定义规则。与 LLM 共同演进。
```

## 目录结构

```text
knowledge/
├─ raw/            # 原始来源文件（不可变，LLM 只读）
├─ converted/      # 转换成品（<tool>/<task-id>/，含质检结论）
├─ sources/        # 来源摘要（简化评分：一手/二手/待验证 + 一句话理由）
├─ wiki/           # 【知识本体】LLM 维护的知识页面
│  ├─ index.md     #   分类目录（entity/concept/synthesis）
│  ├─ log.md       #   追加式操作日志
│  └─ pages/       #   知识页（一个主题一页，持续更新）
├─ domains/        # 领域元信息（可选，不作为入库门槛）
├─ tasks/          # 任务状态
├─ reports/        # 查询报告
├─ templates/      # 模板
└─ inbox/          # 飞书临时落地
```

## 知识页（wiki/pages/）

知识库只有一种原子单元：**知识页**。一个主题一页，持续更新。

### 知识页格式

```yaml
---
id: page-<slug>
title: <标题>
type: entity | concept | synthesis
tags: [tag1, tag2]
sources: [src-xxx, src-yyy]
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: draft | reviewed | stale
---

# <标题>

## 概述
一句话说清这是什么。

## 核心内容
结构化知识。每次加新源时在此补充、修正或标注矛盾。

## 来源
- src-xxx（一手）：<具体出处>
- src-yyy（二手）：<具体出处>

## 关联页面
- [[page-xxx]]：<关联说明>
- [[page-yyy]]：<关联说明>

## 限制与争议
- 标注矛盾、版本差异、覆盖盲区
```

### 页面类型

| 类型 | 说明 | 例子 |
|------|------|------|
| `entity` | 具体实体（人/窑口/器物/材料）| 汝窑、高岭土、唐英 |
| `concept` | 抽象概念（工艺/技术/理论）| 还原焰、釉下彩、玻璃相 |
| `synthesis` | 聚合页（串联多页的导航入口）| 中国陶瓷全览、彩绘演进史 |

> 不再区分"依据卡"和"知识条目"。一页就是一个主题，来源标注于页内。

## 三大工作流

### 1. Ingest（收录）

每次加新源时：

1. 判断：新源涉及哪些已有知识页？需要新建哪些页？
2. **更新已有页**：在相关页面补充信息、修正旧信息、标注矛盾
3. **新建页**：为主题尚不存在的知识新建页面
4. 创建/更新来源摘要（简化评分）
5. 更新 `wiki/index.md`
6. 追加 `wiki/log.md`
7. **git commit**：`ingest: <来源标题> -- <触达页面数> pages`

> 一个来源可能触及 10-15 个已有页面。不要只建新页，要刷新旧页。

### 2. Query（查询）

1. 读 `wiki/index.md` 找相关页面
2. 直接匹配 → 读相关页面 → 综合回答
3. 无直接匹配 → **编织模式**：扫描全索引 → 定位碎片页面 → 逐页阅读 → 合成答案
4. **每个断言必须标注来源页面**。编织出的新结论若综合 ≥3 页且有新洞察 → 建议保存为 synthesis 页
5. 回答结果按专业回答协议输出（问题界定 → 结论等级 → 核心回答 → 依据链 → 限制与反例 → 缺口与下一步 → 意外发现）

### 回答回填

满足以下条件的查询回答应写回 wiki/pages/：

1. 综合了 ≥3 个不同知识页的内容
2. 形成了新的上位概念或跨源洞察
3. 不只是已有页面的重述

保存格式：

```yaml
---
type: synthesis
derived_from: [page-id-1, page-id-2, page-id-3]
---
```

git commit 格式：`synthesis: <页面标题> -- from N sources`

### 3. Lint（认知审计）

定期触发（或每次 Ingest 后自动），检查七个维度：

- [ ] **新鲜度**：confidence:high + last_verified > 90 天的页面 → 标记需复核
- [ ] **暗概念**：wikilinks 引用 ≥3 次但无独立页面的概念 → 提议建新页
- [ ] **矛盾**：两个页面是否对同一事实有冲突说法 → 列出矛盾对，不消解
- [ ] **断链**：[[双括号引用]] 指向不存在的页面
- [ ] **孤立页**：无人引用的页面
- [ ] **覆盖报告**：统计用户查询频率最高的领域 vs 源分布最薄的领域 → 识别薄弱区
- [ ] **缺口感知**：重要概念被提到但没有独立页面；源中提及但 Wiki 无覆盖的主题

## 来源评分（简化版）

| 等级 | 含义 | 例子 |
|------|------|------|
| **一手** | 原始文献、官方文档、考古报告、一手数据 | 窑址发掘报告、清宫档案 |
| **二手** | 研究专著、整理著作、评测、社区整理 | 有泉《陶瓷简史》 |
| **待验证** | 网页、聊天、出处不明的说法 | 论坛帖子、口述 |

> 评分只需等级 + 一句话理由。不再使用六维打分制。来源标注在每页的 sources 字段中。

## 书的长篇处理

### 逐章 Ingest

300 页以上的书不全文一次性 Ingest。流程：

```text
第一轮：只喂目录 → LLM 建立结构页 sources/<书名>.md
        （含章节目录、主题分布、与已有知识页的关联预估）

第二轮起：逐章或分批 Ingest（每次 2-5 章）
  → LLM 读每章 → 提取核心论点 → 更新受影响页面
  → 大部分工作是更新已有页面，不是每章建新页
```

### 300 本书分层策略

| 层级 | 数量 | 处理方式 | 示例 |
|------|------|---------|------|
| **骨架层** | 10-20 本 | 逐章对话式 Ingest，人全程参与 | 《东京梦华录》《宋史》 |
| **填充层** | 50-100 本 | 喂目录+外部摘要，LLM 更新已有页面 | 相关研究专著 |
| **标注层** | 剩余核心 | 只记书名+一句话到 index.md | 边缘相关著作 |
| **待深入层** | 其余 | 不处理，等需要时再说 | 暂未判断价值的书 |

第一本书最慢（建骨架）。第十本书很快——大部分是更新已有页面的某段 + 标注第 N 个来源也支持此观点。

| 文档类型 | 工具 |
|----------|------|
| 扫描书 / 古籍影印 | pdf-craft |
| 复杂版面 / 表格密集 | MinerU |
| 带文本层 PDF | 先抽文本，判断是否需 OCR |

转换记录保留在 `converted/<tool>/<task-id>/`。

## 归档回答

满足以下条件之一的查询回答，应写回 wiki：

1. 回答涉及了 3 个以上知识页的交叉综合
2. 回答补充了新的分析或连接点
3. 用户明确表示有价值

归档方式：新建 synthesis 页，或更新相关 concept 页。

## Schema 演进

AGENTS.md / KNOWLEDGE.md / SOUL.md / RULES.md 不是一成不变的蓝图，而是在对话中长出来的规则集合。

### 生长机制

```text
LLM 在 Ingest/Query/Lint 中遇到 Schema 未覆盖的情况
  → LLM 主动报告：
      「我注意到 XX 情况在现有 Schema 里没有规则。
        建议在 YY.md 中加入：<规则草案>。
        原因：<实际案例>。要不要加？」
  → boss 确认 → LLM 修改对应 Schema 文件
  → git commit: 「schema: YY.md -- XX 规则 (trigger: <触发原因>)」
```

**硬约束**：LLM 不能自行修改 Schema。必须经过提议 → 确认 → 执行。

### 纪录格式

每条新规则写入时附带简短注释说明触发原因，避免未来忘记「当时为什么加这条」。

## 有机目录演化

wiki/pages/ 初始为平铺。用 frontmatter 的 `type: entity | concept | synthesis` 区分页面类型，而非子目录。

### 演化规则

- 当某 type 的页面数超过 10 → LLM 在 Lint 或 Ingest 中主动提议：
  - 选项 A：建子目录（如 wiki/pages/concepts/）
  - 选项 B：建导航索引页（如 concept-index.md）
- boss 决定后 → LLM 执行迁移（移动文件 + 更新所有 [[wikilinks]]）
- 迁移操作必须 git commit 以便回滚

### 当前状态

wiki/pages/ 为平铺结构。尚未积累到触发子目录演化的阈值。
